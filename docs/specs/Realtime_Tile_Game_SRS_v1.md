# 1. 문서 목적

본 문서는 학원 내부망에서 2~4명의 사용자가 함께 플레이할 수 있는 실시간 숫자 타일 보드게임의 기능, 비기능, 데이터, 통신, 보안, 예외 처리 및 인수 조건을 정의한다.

본 문서의 목적은 다음과 같다.

- 기능 구현 범위를 명확히 고정한다.
- 프론트엔드와 백엔드의 책임을 분리한다.
- 게임 규칙 판정 기준을 서버 중심으로 통일한다.
- 실시간 상태 동기화와 재접속 정책을 명확히 한다.
- 테스트 가능한 요구사항과 완료 기준을 정의한다.
- 포트폴리오에서 사용자 본인의 기획·검증·품질관리 역할을 증명할 수 있게 한다.

# 2. 프로젝트 개요

## 2-1. 서비스 한 줄 소개

2~4명의 사용자가 방을 만들고 실시간으로 숫자 타일을 배치하며 그룹과 연속수를 완성하는 웹 기반 멀티플레이 보드게임 서비스.

## 2-2. 핵심 가치

- 단순 CRUD를 넘어선 실시간 멀티플레이 경험
- 서버 권한형 게임 상태 관리
- 비공개 손패와 공개 테이블의 분리
- 복잡한 규칙 검증 엔진
- 턴 단위 임시 상태와 원자적 롤백
- 시간 초과 및 재접속 복구
- 2~4인 실제 플레이 가능

## 2-3. 저작권 및 리소스 정책

본 프로젝트는 기존 상용 루미큐브 서비스의 다음 요소를 복제하지 않는다.

- 로고
- 브랜드명
- 타일 이미지
- 배경 이미지
- UI 레이아웃
- 사운드
- 기타 고유 시각 리소스

숫자 타일 기반 게임 규칙은 참고하되, 프로젝트명과 디자인은 자체 제작한다.

# 5. 기능 요구사항

## FR-AUTH-001 회원가입

시스템은 이메일, 비밀번호, 비밀번호 확인, 닉네임을 입력받아 사용자를 생성해야 한다.

인수 조건:

- 이메일 형식이 유효해야 한다.
- 이메일은 중복될 수 없다.
- 닉네임은 중복될 수 없다.
- 비밀번호와 비밀번호 확인이 일치해야 한다.
- 서버 요청 중 중복 제출이 차단되어야 한다.
- 비밀번호는 평문으로 저장하면 안 된다.

## FR-AUTH-002 로그인

시스템은 이메일과 비밀번호를 검증하고 인증 상태를 생성해야 한다.

인수 조건:

- 성공 시 로비 또는 재접속 화면으로 이동한다.
- 실패 시 원인을 사용자 친화적인 메시지로 표시한다.
- Access Token은 클라이언트 메모리 중심으로 관리한다.
- Refresh Token 구조는 HttpOnly Cookie 사용을 권장한다.

## FR-AUTH-003 로그아웃

시스템은 현재 인증 상태와 Refresh Token을 제거해야 한다.

## FR-PROFILE-001 프로필 설정

사용자는 닉네임과 기본 아바타를 설정할 수 있어야 한다.

---

## FR-LOBBY-001 방 목록 조회

사용자는 현재 입장 가능한 방 목록을 확인할 수 있어야 한다.

표시 정보:

- 방 이름
- 방장 닉네임
- 현재 인원
- 최대 인원
- 방 상태
- 입장 가능 여부

## FR-LOBBY-002 방 목록 실시간 갱신

방 생성, 삭제, 인원 변경, 게임 시작이 발생하면 로비 목록이 갱신되어야 한다.

## FR-ROOM-001 방 생성

사용자는 다음 정보를 입력해 방을 생성할 수 있어야 한다.

- 방 이름
- 최대 인원 2, 3, 4 중 하나
- 턴 제한시간
- 공개 여부

기본값:

```text
최대 인원: 4
턴 제한시간: 120초
공개방
```

## FR-ROOM-002 방 입장

사용자는 대기 중이며 정원이 남은 방에 입장할 수 있어야 한다.

입장 실패 조건:

- 게임 중
- 방 종료
- 정원 초과
- 이미 다른 대기방 또는 게임 참가 중
- 인증되지 않은 사용자

## FR-ROOM-003 방 나가기

대기방 사용자는 방을 나갈 수 있어야 한다.

방장이 나가면 다음 입장 순서 사용자에게 방장을 위임해야 한다.

## FR-ROOM-004 준비 상태

참가자는 준비 상태를 변경할 수 있어야 한다.

## FR-ROOM-005 게임 시작

방장만 게임 시작 요청을 보낼 수 있다.

시작 조건:

- 참가자 2명 이상
- 참가자 4명 이하
- 모든 참가자 준비 완료
- 방 상태가 WAITING
- 게임이 이미 생성되지 않음

## FR-GAME-001 타일 세트 생성

서버는 게임마다 106개의 고유 타일을 생성해야 한다.

구성:

```text
4색 × 1~13 × 2세트 = 104개
조커 = 2개
```

각 타일은 고유 tileId를 가져야 한다.

## FR-GAME-002 타일 셔플 및 분배

서버는 타일을 무작위로 섞고 참가자에게 각 14개씩 분배해야 한다.

남은 타일은 공용 풀에 저장한다.

## FR-GAME-003 선 플레이어 결정

서버는 참가자 중 한 명을 무작위로 선 플레이어로 결정해야 한다.

## FR-GAME-004 턴 순환

턴은 방 입장 순서를 기준으로 반복되어야 한다.

```text
2인: P1 → P2 → P1
3인: P1 → P2 → P3 → P1
4인: P1 → P2 → P3 → P4 → P1
```

## FR-GAME-005 손패 비공개

각 사용자는 자신의 손패 전체 정보만 받을 수 있어야 한다.

다른 사용자는 다음만 확인할 수 있다.

- 닉네임
- 아바타
- 손패 개수
- 첫 등록 상태
- 현재 턴 여부
- 연결 상태

상대 손패의 tileId, 색상, 숫자, 정렬 순서는 전송하면 안 된다.

## FR-RULE-001 RUN 검증

RUN은 같은 색상의 연속된 숫자 3개 이상이어야 한다.

검증 조건:

- 최소 3개
- 같은 색상
- 1씩 증가
- 숫자 중복 없음
- 13 다음 1 연결 금지
- 최대 13개

## FR-RULE-002 GROUP 검증

GROUP은 같은 숫자의 서로 다른 색상 타일 3개 또는 4개여야 한다.

검증 조건:

- 길이 3 또는 4
- 숫자 동일
- 색상 중복 금지

## FR-RULE-003 첫 등록

첫 등록 전 플레이어는 자신의 손패만 사용해 합계 30점 이상의 유효 조합을 제출해야 한다.

첫 등록 제한:

- 기존 테이블 타일 사용 금지
- 기존 테이블 조작 금지
- 여러 유효 조합 합산 가능
- 조커 사용 가능
- 조커 점수는 대체 숫자
- 성공한 첫 등록 턴에도 기존 테이블 조작 금지

## FR-RULE-004 일반 턴 기여 조건

첫 등록 완료 후 턴을 확정하려면 손패에서 최소 1개 이상의 타일이 테이블로 이동해야 한다.

기존 테이블 타일만 재배열하는 행동은 확정할 수 없다.

## FR-RULE-005 테이블 전체 유효성

턴 확정 시 테이블 위 모든 타일은 중복이나 유실 없이 하나의 유효 RUN 또는 GROUP에 포함되어야 한다.

## FR-RULE-006 기존 테이블 재조합

첫 등록을 완료한 사용자는 기존 테이블 조합을 분리, 병합, 이동할 수 있어야 한다.

단:

- 테이블 타일을 손패로 가져갈 수 없다.
- 최종 테이블 전체가 유효해야 한다.
- 손패 타일을 최소 1개 사용해야 한다.

## FR-RULE-007 조커 사용

조커는 유효 조합에 필요한 일반 타일 하나를 대신할 수 있어야 한다.

## FR-RULE-008 조커 회수

첫 등록 완료 후 실제 대체 타일로 조커를 교체할 수 있어야 한다.

회수한 조커는 같은 턴에 새로운 유효 조합에 반드시 사용해야 한다.

조커를 손패로 저장한 채 턴을 종료할 수 없다.

## FR-TURN-001 턴 시작 스냅샷

서버는 매 턴 시작 시 다음 상태를 스냅샷으로 저장해야 한다.

- 공개 테이블
- 현재 플레이어 손패
- 타일 위치
- 공용 풀
- 첫 등록 상태
- 현재 플레이어
- gameVersion

## FR-TURN-002 TurnDraft

사용자의 임시 배치는 확정된 GameState와 분리된 TurnDraft로 관리해야 한다.

## FR-TURN-003 턴 취소

사용자는 현재 턴의 임시 배치를 취소할 수 있어야 한다.

취소 시 턴 시작 스냅샷 상태로 복원하고 턴은 유지한다.

## FR-TURN-004 턴 확정

서버는 턴 확정 요청을 검증한 뒤 성공할 때만 GameState에 반영해야 한다.

검증 실패 시 확정 GameState는 변경되면 안 된다.

## FR-TURN-005 타일 뽑기

사용자는 공용 풀에서 타일 1개를 뽑을 수 있어야 한다.

조건:

- 임시 배치가 있다면 먼저 취소 확인
- 턴 시작 상태 복원
- 타일 1개 드로우
- 뽑은 타일은 같은 턴 사용 불가
- 즉시 다음 턴으로 전환

## FR-TURN-006 PASS

공용 풀이 비어 있을 때만 PASS가 가능해야 한다.

## FR-TIMER-001 서버 기준 타이머

턴 제한시간의 최종 판정은 서버 시간이 기준이어야 한다.

클라이언트 타이머는 표시 목적으로만 사용한다.

## FR-TIMER-002 시간 초과 복원

시간 초과 시 서버는 다음을 하나의 원자적 처리로 수행해야 한다.

```text
TurnDraft 폐기
→ 턴 시작 스냅샷 복원
→ 손패·테이블·조커 위치 복원
→ 공용 풀이 남아 있으면 1개 자동 드로우
→ 풀이 비어 있으면 PASS
→ 다음 플레이어 턴
```

유효한 조합을 일부 만들어 두었더라도 확정하지 않았다면 모두 취소해야 한다.

## FR-TIMER-003 중복 시간 초과 방지

시간 초과 처리는 한 턴당 정확히 한 번만 실행되어야 한다.

## FR-SORT-001 777 정렬

모든 클라이언트는 동일한 확정 GameState 버전을 공유해야 한다.

## NFR-RECOVERY-001 원자성

검증 실패, 서버 오류, 시간 초과 처리 중 일부만 반영되는 상태를 허용하면 안 된다.

## NFR-SEC-001 인증

게임 변경 요청은 인증 사용자만 수행할 수 있어야 한다.

## NFR-SEC-002 권한

사용자는 자신이 참가한 방과 게임에 대해서만 요청할 수 있어야 한다.

## NFR-SEC-003 손패 기밀성

상대 손패 원문은 API 응답, WebSocket 이벤트, 로그, 프론트 상태에 노출하면 안 된다.

## NFR-SEC-004 입력 신뢰 금지

클라이언트가 보낸 tileId, gameVersion, playerId, roomId를 서버가 재검증해야 한다.

## NFR-UX-001 중복 입력 방지

서버 요청 진행 중 관련 버튼을 비활성화해야 한다.

## NFR-UX-002 오류 안내

오류 코드는 사용자 친화적 한국어 메시지로 변환해야 한다.

## NFR-COMPAT-001 화면 해상도

- 1366×768

# 13. 실제 시나리오 인수 테스트

## AT-001 2인 기본 게임 시작

```text
사용자 A 로그인
→ 방 생성, 최대 인원 2
→ 사용자 B 입장
→ 두 사용자 준비
→ 방장 시작
→ 두 사용자에게 14개 손패 분배
→ 현재 턴 표시
```

통과 기준:

- 참가자 2명으로 정상 시작
- 손패는 각자 자기 것만 확인
- 공용 풀 수량 정확

## AT-002 4인 게임 시작

4명의 참가자가 입장하고 준비 완료 후 정상 시작해야 한다.

## AT-003 첫 등록

30점 이상의 조합은 성공하고 29점은 실패해야 한다.

## AT-004 777/789 정렬

정렬 버튼은 서버 상태를 변경하지 않으면서 화면 순서만 변경해야 한다.

## AT-005 잘못된 배치 시간 초과

```text
턴 시작
→ 사용자가 잘못된 조합 배치
→ 확정하지 않고 시간 초과
→ 배치 전 손패와 테이블 복원
→ 자동 드로우 1회
→ 다음 사용자 턴
```

통과 기준:

- 타일 중복 없음
- 타일 유실 없음
- 조커 위치 복원
- 다른 사용자 화면도 동일
- 자동 드로우 1회만 발생

## AT-006 중복 턴 확정

확정 버튼 연속 클릭 또는 동일 actionId 재전송 시 한 번만 처리되어야 한다.

## AT-007 상대 손패 보호

브라우저 개발자 도구와 WebSocket 수신 메시지에서 상대 타일 원문이 확인되지 않아야 한다.

## AT-008 재접속

게임 중 새로고침 후 기존 게임으로 복귀하며 자신의 손패와 최신 테이블이 복원되어야 한다.

## AT-009 조커 회수

정상 타일로 조커를 교체하고 같은 턴에 새 유효 조합에 사용해야 한다.

## AT-010 게임 종료

손패 0개 확정 후 게임이 종료되고 추가 요청이 차단되어야 한다.

# 17. 게임 모드·랭킹·상대 전적 추가 요구사항

## FR-MODE-001 게임 모드 선택

방 생성자는 `CLASSIC` 또는 `SPEED`를 선택할 수 있어야 한다.

## FR-MODE-002 CLASSIC

CLASSIC은 기존 첫 등록 30점, 손패 0개 승리, 교착 종료 규칙을 사용해야 한다.

CLASSIC 결과는 랭킹 점수에 반영해야 한다.

## FR-MODE-003 SPEED

SPEED는 다음 규칙을 사용해야 한다.

- 전체 게임 제한 5분
- 기본 턴 제한 20초
- 첫 등록 30점 미적용
- 제한시간 종료 후 점수 승부
- 랭킹 미반영

## FR-MODE-004 SPEED 점수

```text
최종 점수
= 자신의 손패에서 테이블에 확정한 타일 점수
- 종료 시 남은 손패 점수
```

## FR-MODE-005 DRAW

최종 승리 조건에서 동점자가 둘 이상이면 `DRAW` 처리해야 한다.

---

## FR-RATING-001 초기 랭킹

신규 사용자의 초기 랭킹 점수는 1000점이어야 한다.

## FR-RATING-002 반영 모드

CLASSIC 결과만 랭킹에 반영해야 한다.

## FR-RATING-003 상대 평균

각 플레이어의 상대 기준 점수는 자신을 제외한 참가자의 경기 시작 전 평균 랭킹이어야 한다.

## FR-RATING-004 점수 계산

권장 계산식:

```text
expectedScore = 1 / (1 + 10 ^ ((opponentAverageRating - playerRating) / 400))
ratingDelta = round(32 × (actualScore - expectedScore))
```

```text
승리 = 1.0
DRAW = 0.5
패배 = 0.0
```

## FR-RATING-005 원자성

게임 결과 저장과 랭킹 점수 반영은 중복 없이 한 번만 실행되어야 한다.

## FR-RATING-006 최소 점수

랭킹 점수는 0 미만이 될 수 없다.

---

## FR-RECORD-001 상대 전적

사용자는 특정 상대와의 전적을 확인할 수 있어야 한다.

표시 항목:

- 총 대전 수
- 모드별 승
- 모드별 패
- 모드별 무
- 전체 승률
- 최근 대전일

## FR-RECORD-002 결과 기록

모든 완료된 CLASSIC 및 SPEED 게임은 참가자 간 상대 전적에 반영해야 한다.

## FR-RECORD-003 중복 방지

하나의 게임 결과가 상대 전적에 두 번 반영되면 안 된다.

## 추가 인수 테스트

### AT-011 낮은 상대 평균에게 승리

자신보다 낮은 평균 랭킹의 상대에게 승리했을 때 비슷하거나 높은 상대에게 이겼을 때보다 획득 점수가 작아야 한다.

### AT-012 높은 상대 평균에게 승리

자신보다 높은 평균 랭킹의 상대에게 승리했을 때 낮은 상대에게 이겼을 때보다 획득 점수가 커야 한다.

### AT-013 DRAW

동점 결과는 `DRAW`로 저장되고 actualScore 0.5로 계산되어야 한다.

### AT-014 SPEED 랭킹 미반영

SPEED 경기 종료 후 사용자 랭킹 점수가 변경되면 안 된다.

### AT-015 상대 전적

두 사용자가 게임을 마치면 양쪽 사용자 관점의 승·패·무가 반대로 정확히 기록되어야 한다.

### AT-016 랭킹 중복 반영 방지

같은 게임 종료 이벤트가 재전송되어도 랭킹과 상대 전적은 한 번만 반영되어야 한다.
