# 1. 프로젝트 최종 방향

이 프로젝트의 1차 목표는 2~4인 실시간 숫자 타일 보드게임을 완성하는 것이다.

하지만 코드 구조는 단일 게임에서 끝나지 않고, 장기적으로 다음 형태의 **보드게임 플랫폼**으로 확장할 수 있게 설계한다.

```text
사용자
→ 로비
→ 게임방
→ 게임 모드 선택
→ 게임 세션
→ 결과
→ 전적
→ 랭킹
```

후속 확장 예:

```text
숫자 타일 게임
카드 게임
턴제 보드게임
팀전
관전자
토너먼트
시즌 랭킹
친선전
사용자 제작 룰
```

현재 기능에 필요하지 않은 플랫폼 기능을 미리 과도하게 구현하지 않는다.

대신 다음 확장 지점은 처음부터 고려한다.

```text
GameType
GameMode
RulePolicy
GameSession
GameResult
RatingPolicy
MatchHistory
PlayerSnapshot
PublicState / PrivateState
```

# 2. 코드 품질 원칙

## 2-1. 시니어 개발자가 작성한 코드 수준을 지향한다

단순히 동작하는 코드를 목표로 하지 않는다.

다음을 만족하는 코드를 작성한다.

- 책임이 명확한 클래스
- 짧고 의미 있는 메서드
- 의도를 드러내는 이름
- 중복 로직 제거
- 상태 변경 지점 최소화
- 테스트 가능한 구조
- 확장 가능한 정책 분리
- 예외 상황을 숨기지 않는 코드
- 읽는 사람이 흐름을 추적할 수 있는 구조

금지:

```text
거대한 Service 한 개에 모든 로직 집중
Controller에서 게임 규칙 처리
WebSocket Handler에서 직접 상태 수정
DTO를 Domain 객체처럼 사용
JPA Entity에 모든 비즈니스 규칙 집중
boolean 플래그와 if문만으로 모드 확장
의미 없는 Utils 클래스 남발
```

# 3. 정합성과 무결성을 최우선으로 본다

기능 수보다 게임 상태의 정확성을 우선한다.

핵심 불변조건:

```text
모든 106개 tileId는 정확히 한 위치에 존재한다.
동일 tileId는 중복될 수 없다.
tileId는 유실될 수 없다.
확정된 테이블은 모두 유효한 조합이어야 한다.
현재 턴 사용자만 상태를 변경할 수 있다.
실패한 요청은 확정 상태를 변경하지 않는다.
상대 손패 원문은 다른 사용자에게 전송하지 않는다.
```

새 기능을 구현할 때마다 다음을 확인한다.

```text
이 기능이 타일 중복을 만들 수 있는가?
이 기능이 상태 일부만 반영될 가능성이 있는가?
이 기능이 오래된 클라이언트 요청을 허용하는가?
이 기능이 다른 사용자의 비공개 정보를 노출하는가?
이 기능이 중복 요청에 의해 두 번 실행될 수 있는가?
```

# 4. 원자성을 최우선으로 본다

상태 변경은 전부 성공하거나 전부 실패해야 한다.

권장 흐름:

```text
현재 상태 조회
→ 복사본 또는 Candidate 생성
→ 전체 검증
→ 불변조건 검증
→ 성공 시 한 번에 Commit
```

금지:

```text
현재 GameState 일부 수정
→ 중간 검증 실패
→ 수동으로 일부 되돌리기
```

필수 적용 대상:

- 턴 확정
- 타일 드로우
- 시간 초과
- 조커 교체
- 테이블 재조합
- 게임 종료
- 랭킹 반영
- 상대 전적 반영

시간 초과 처리:

```text
TurnSnapshot 복원
→ 자동 드로우 또는 PASS
→ 다음 턴 생성
```

전체가 하나의 원자적 처리여야 한다.

# 8. 동시성과 중복 요청을 기본 전제로 본다

사용자는 버튼을 연속 클릭할 수 있다.

네트워크는 요청을 재전송할 수 있다.

서로 다른 사용자가 동시에 행동할 수 있다.

따라서 다음을 기본으로 적용한다.

```text
게임 ID별 단일 상태 변경 Lock
actionId 중복 방지
gameVersion 낙관적 버전 확인
TurnState.resolved 중복 종료 방지
DB UNIQUE 제약
```

메모리에서 막았더라도 DB 제약으로 한 번 더 보호한다.

# 11. 과도한 추상화를 금지한다

보드게임 플랫폼 확장을 고려하되, 현재 두 번째 게임이 존재하지 않는 영역은 성급하게 추상화하지 않는다.

새 추상화는 다음 중 하나가 있을 때만 도입한다.

```text
실제 중복 구현이 두 곳 이상 존재
두 번째 게임 구현 계획이 구체적
현재 코드의 변경 비용을 명확히 줄임
```

금지:

```text
막연한 미래 가능성만으로 인터페이스 추가
한 개 구현체만 존재하는 의미 없는 Factory
추상 클래스와 Strategy의 중복 계층
범용성을 이유로 현재 요구사항을 이해하기 어렵게 만드는 구조
```

# 12. REST API·WebSocket·public 계약 호환성

다음 변경은 일반 리팩터링으로 취급하지 않는다.

```text
REST API 요청·응답 필드 변경
WebSocket destination 변경
WebSocket payload 변경
public enum 변경
오류 코드 변경
DB 컬럼 의미 변경
```

원칙:

- 기존 필드 삭제보다 신규 필드 추가를 우선한다.
- 필드 이름 변경 전 프론트·백엔드·테스트 사용처를 확인한다.
- `gameVersion`, `actionId`, 오류 코드의 의미를 함부로 변경하지 않는다.
- 호환성을 깨는 변경은 별도 작업지시서와 마이그레이션 절차를 만든다.
- API와 WebSocket 문서를 함께 갱신한다.

# 13. DB 설계와 마이그레이션 원칙

- 게임 중 자주 바뀌는 상태는 우선 메모리에서 관리한다.
- 결과, 전적, 랭킹, 사용자, 방 정보는 DB에 저장한다.
- JPA 연관관계는 기본 LAZY를 사용한다.
- 조회는 DTO로 분리한다.
- N+1을 확인한다.
- 페이징과 컬렉션 fetch join을 무리하게 함께 쓰지 않는다.
- 중복 반영은 UNIQUE 제약으로 보호한다.
- 결과와 랭킹 반영은 하나의 트랜잭션으로 묶는다.

DB 스키마 변경은 Flyway 등 마이그레이션 파일로 관리한다.

```text
V1__create_users.sql
V2__create_rooms.sql
V3__add_rating_score.sql
```

원칙:

- 이미 적용된 마이그레이션 파일을 수정하지 않는다.
- 새 변경은 새 버전 파일로 추가한다.
- 컬럼 삭제·타입 변경은 데이터 손실 가능성을 검토한다.
- 스키마 변경과 코드 배포 순서를 문서화한다.
- 운영 기준에서 `ddl-auto=update`에 의존하지 않는다.
- 개발 초기에는 `validate` 또는 명시적 마이그레이션을 우선한다.

금지:

```text
게임 드래그마다 DB UPDATE
전체 손패를 로그에 기록
무제한 양방향 연관관계
Entity 직접 API 응답
적용된 마이그레이션 파일 수정
```

# 16. 성능과 부하 기준

한 게임의 상태 변경은 직렬화하되, 서로 다른 게임은 병렬로 실행할 수 있어야 한다.

원칙:

- 게임 Lock 안에서 DB 조회를 최소화한다.
- 게임 Lock 안에서 WebSocket 전송을 직접 오래 수행하지 않는다.
- Snapshot 생성과 규칙 검증 시간을 측정할 수 있게 한다.
- 전체 테이블 검증은 최대 106개 타일 규모에서 정확성을 우선한다.
- 불필요한 조기 최적화를 하지 않는다.

구조 검증용 최소 목표:

```text
동시 게임방 10개
게임당 4명
동시 WebSocket 연결 40개
```

이는 상용 SLA가 아니라 내부 구조 검증 기준이다.

# 17. 장애 처리와 Commit·DB·이벤트 경계

다음 세 경계를 구분한다.

```text
메모리 GameState Commit
DB 결과 저장
WebSocket 이벤트 전송
```

원칙:

- WebSocket 전송 실패 때문에 이미 확정된 GameState를 임의로 되돌리지 않는다.
- 클라이언트는 재접속 시 최신 Snapshot으로 복구한다.
- 게임 종료 결과 DB 저장 실패 시 랭킹과 상대 전적을 부분 반영하지 않는다.
- 결과·랭킹·전적은 하나의 DB 트랜잭션으로 묶는다.
- 부수효과 실패를 숨기지 않고 로그와 재처리 가능 상태로 남긴다.
- 1차에서는 복잡한 Outbox 인프라를 강제하지 않지만, 도메인 상태와 외부 전송 책임은 분리한다.

# 18. 스케줄러 소유권과 중복 실행 방지

각 Turn은 고유 `turnId`를 가진다.

타이머 실행 시 반드시 다음을 검증한다.

```text
gameId 일치
turnId 일치
현재 Turn과 동일
resolved == false
현재 서버 시각 >= deadlineAt
```

원칙:

- 이전 턴의 늦게 실행된 timeout은 상태를 변경하지 않는다.
- 같은 턴 timeout은 한 번만 처리한다.
- 게임 종료 시 예약된 타이머를 정리한다.
- SPEED 전체 게임 타이머와 턴 타이머가 충돌하면 전체 게임 종료를 우선한다.
- 스케줄러 콜백은 현재 상태를 다시 조회한 뒤 처리한다.

이 항목은 P0 검증 대상이다.

# 20. 테스트 우선순위

테스트는 기능 완료의 일부다.

최소 구성:

```text
Unit
Integration
REST API
WebSocket
E2E
```

P0 우선 검증:

- 타일 중복·유실
- 상태 일부 반영
- 상대 손패 노출
- 중복 요청
- 오래된 요청
- 시간 초과 중복 처리
- 게임 결과 중복 저장
- 랭킹·전적 중복 반영
- 오래된 스케줄러 작업 실행
- DB 마이그레이션 실패

코드가 동작하더라도 핵심 실패 테스트가 없으면 완료로 보지 않는다.

# 21. diff 기반 코드 수정 원칙

기존 코드 수정은 전체 파일 재작성보다 diff 기반으로 진행한다.

기본 산출물:

```text
수정·생성 파일 목록
변경 이유
patch ZIP
자동 테스트
직접 확인 절차
완료 보고서 md
```

금지:

```text
매 작업마다 전체 프로젝트 ZIP 재생성
요청 범위와 무관한 파일 수정
포맷팅만으로 대규모 diff 생성
파일 전체를 이유 없이 교체
```

전체 프로젝트 ZIP은 다음 경우에만 사용한다.

- 최초 프로젝트 전달
- 큰 구조 감사
- 패치 적용 후 전체 검수
- 기준점 백업

코드 구현 결과는 수정·생성 파일만 포함한 패치 ZIP을 우선한다.

# 22. 변경 범위를 통제한다

한 작업에서는 하나의 목적만 해결한다.

예:

```text
Phase 1 타일 생성
Phase 1 RUN 검증
Phase 1 GROUP 검증
```

처럼 위험도가 높으면 더 작게 나눈다.

다음은 별도 작업으로 분리한다.

- public API 변경
- DB 스키마 변경
- 저장·복원 포맷 변경
- WebSocket 메시지 변경
- 핵심 도메인 책임 이동
- 랭킹 공식 변경

무관한 리팩터링은 하지 않는다.

필요한 기술부채는 문서에 기록하고 후속 작업으로 남긴다.

# 23. Definition of Ready

작업 시작 전 다음이 확인되어야 한다.

```text
목적이 한 문장으로 설명됨
입력과 출력이 정의됨
성공 조건이 정의됨
실패 조건이 정의됨
영향을 받는 API·메시지·DB·상태가 확인됨
테스트 ID가 정해짐
호환성 변경 여부가 확인됨
롤백 또는 실패 시 상태가 정의됨
```

이 조건을 충족하지 못하면 코드를 먼저 작성하지 않는다.

# 24. 기존 동작 보존과 회귀 검증

새 기능이 기존 기능을 깨지 않아야 한다.

모든 작업 후:

```text
신규 테스트
기존 테스트
대표 E2E
```

를 확인한다.

버그 수정 시 반드시 재현 테스트를 먼저 추가하거나, 최소한 동일 실패를 다시 검증할 수 있는 자동 테스트를 남긴다.

# 25. 의존성·버전 관리

새 라이브러리는 기존 기능으로 해결하기 어려울 때만 추가한다.

원칙:

- 추가 이유 기록
- 대체안 검토
- 중복 역할 라이브러리 동시 사용 금지
- 버전 명시
- 취약점과 유지 상태 확인
- 삭제 가능한 의존성은 제거

AI가 제안했다는 이유만으로 라이브러리를 추가하지 않는다.

# 26. 코드 주석 기준

코드가 무엇을 하는지는 이름과 구조로 표현한다.

주석은 다음을 설명한다.

```text
왜 이 정책을 택했는가
왜 이 순서가 필요한가
어떤 무결성을 보호하는가
어떤 과거 버그를 막는가
```

금지:

```text
코드와 같은 내용을 반복하는 주석
오래된 주석
설명 없는 TODO
```

TODO는 이유와 제거 조건을 기록한다.

예:

```java
// 같은 actionId 재전송 시 랭킹이 중복 반영되지 않도록
// 게임 종료 트랜잭션 안에서 처리 여부를 먼저 확인한다.
```

# 28. 시간 처리 원칙

게임 시간 판정은 서버 시간을 기준으로 한다.

클라이언트 타이머:

```text
표시용
```

```text
최종 판정
```

SPEED에서는 전체 게임 시간과 턴 시간이 충돌하면 전체 게임 종료를 우선한다.

```text
GameDeadline > TurnDeadline
```

# 29. 프론트엔드 원칙

- Pinia Store 책임을 분리한다.
- 공개 상태와 개인 상태를 분리한다.
- TurnDraft는 확정 GameState와 분리한다.
- 정렬은 로컬 표시 기능으로 유지한다.
- 요청 중 버튼을 비활성화한다.
- 서버 성공 이벤트 전 로컬 확정 상태로 취급하지 않는다.
- 오래된 이벤트를 gameVersion으로 무시한다.
- 버전 누락이 의심되면 전체 상태를 재조회한다.

권장 Store:

```text
authStore
lobbyStore
roomStore
gamePublicStore
gamePrivateStore
turnDraftStore
connectionStore
```

# 30. 코드 리뷰 기준

모든 코드 검수에서 다음을 확인한다.

## 책임

- 이 클래스가 이 책임을 가져야 하는가?
- 도메인 규칙이 Controller나 Handler에 새어 나오지 않았는가?

## 상태

- 상태를 검증 전에 수정하지 않는가?
- 실패 시 완전히 원상복구되는가?

## 동시성

- 동일 요청이 두 번 실행될 수 있는가?
- 게임 Lock 범위가 적절한가?
- 오래된 타이머가 실행될 수 있는가?

## 보안

- 상대 손패가 노출될 수 있는가?
- 인증 사용자와 payload userId를 혼동하지 않는가?
- 비밀정보가 로그나 저장소에 들어가는가?

## 확장성

- 모드 분기가 여러 곳에 흩어졌는가?
- 플랫폼 공통 영역과 게임 전용 영역이 섞였는가?
- 불필요한 추상화가 추가되었는가?

## 호환성

- REST·WebSocket·DB 계약이 깨지는가?
- 기존 클라이언트와 테스트에 영향이 있는가?

## 테스트

- 성공만 있고 실패 테스트가 없는가?
- 버그 재현 테스트가 있는가?

# 31. 아키텍처 결정 기록

중요한 구조 결정은 ADR 또는 결정 로그로 남긴다.

기록 항목:

```text
문제
선택한 방식
검토한 대안
선택 이유
단점
변경이 필요한 조건
```

예:

```text
ADR-001 GameState를 DB가 아닌 서버 메모리에 둔다.
ADR-002 상대 손패는 개인 WebSocket 채널로만 전송한다.
ADR-003 SPEED는 랭킹에 반영하지 않는다.
ADR-004 게임 ID별 단일 Lock을 사용한다.
```

ADR은 포트폴리오의 기술적 의사결정 근거로 활용한다.

# 33. 포트폴리오 관점

사용자가 직접 코드를 모두 작성하지 않더라도 다음 능력을 증명해야 한다.

- 요구사항 정의
- 시스템 구조 판단
- AI 코드 검수
- 실행 환경 구성
- 오류 재현
- 로그 분석
- 테스트 설계
- 수정 결과 검증
- 회귀 테스트
- 기술적 선택 설명

면접 전에 반드시 설명할 수 있어야 하는 항목:

```text
왜 REST와 WebSocket을 나눴는가
왜 서버 권한형인가
왜 GameState를 메모리에 두는가
왜 actionId와 gameVersion이 둘 다 필요한가
왜 TurnSnapshot이 필요한가
상대 손패를 어떻게 숨기는가
CLASSIC과 SPEED를 어떻게 분리했는가
랭킹 중복 반영을 어떻게 막는가
보드게임 플랫폼으로 어떻게 확장할 수 있는가
왜 과도한 추상화를 피했는가
DB 마이그레이션을 어떻게 관리하는가
장애 시 GameState·DB·이벤트를 어떻게 다루는가
```

# 35. 최종 개발 철학

```text
빠르게 많이 만드는 것보다
작게 만들고 정확하게 검증한다.

동작하는 코드보다
상태가 깨지지 않는 코드를 만든다.

현재 기능만 맞추는 것보다
다음 게임을 추가할 수 있는 경계를 만든다.

과도한 추상화보다
실제 확장 지점을 명확히 분리한다.

전체 파일 교체보다
작은 diff와 검증 가능한 패치를 남긴다.

API와 메시지는 계약으로 다루고,
DB 변경은 마이그레이션으로 남긴다.

장애를 숨기지 않고,
재현·복구·재검증할 수 있게 만든다.
```
