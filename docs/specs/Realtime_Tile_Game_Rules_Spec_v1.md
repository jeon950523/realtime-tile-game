# 실시간 타일 보드게임 게임 규칙 명세 v1

작성 기준: 2026-07-14  
문서 상태: 1차 MVP 구현 기준 확정안  
대상 인원: 2~4인  
참조: Rummikub Classic 공식 규칙을 기반으로 하되, 웹 멀티플레이 구현을 위해 일부 운영 규칙을 명시적으로 고정한다.

> 이 프로젝트는 상용 브랜드, 로고, 이미지, UI 리소스를 복제하지 않는다.  
> 숫자 타일 기반의 자체 명칭·자체 UI 게임으로 제작한다.

---

## 1. 규칙 설계 목표

이 문서는 다음 상황에서 서버가 항상 같은 결론을 내리도록 만드는 것이 목적이다.

- 어떤 타일 조합이 유효한가
- 첫 등록이 성공했는가
- 조커가 어떤 값을 대신하는가
- 기존 테이블을 어떻게 재조합할 수 있는가
- 턴을 확정할 수 있는가
- 실패 시 무엇을 원상복구하는가
- 누가 승리했는가

게임 규칙의 최종 판정 권한은 클라이언트가 아니라 서버가 가진다.

---

## 2. 타일 구성

### 2-1. 일반 타일

색상 4종:

```text
RED
BLUE
YELLOW
BLACK
```

숫자:

```text
1 ~ 13
```

각 `색상 + 숫자` 조합은 2개씩 존재한다.

```text
4 colors × 13 numbers × 2 copies = 104 tiles
```

### 2-2. 조커

```text
JOKER 2개
```

### 2-3. 전체 타일 수

```text
일반 타일 104개 + 조커 2개 = 총 106개
```

### 2-4. 타일 식별자

같은 색상과 숫자의 타일도 실제 객체는 서로 구별해야 한다.

예:

```text
RED-7-A
RED-7-B
JOKER-A
JOKER-B
```

서버는 화면 표시값이 아니라 고유 `tileId`로 소유권과 이동을 검증한다.

---

## 3. 게임 시작

### 3-1. 참가 인원

```text
최소 2명
최대 4명
```

2명 미만이면 게임을 시작할 수 없다.

2인 게임도 정식 지원하며, 턴 순서는 두 플레이어가 번갈아 진행한다.

### 3-2. 시작 조건

- 방의 모든 참가자가 준비 상태여야 한다.
- 방장만 시작 요청을 보낼 수 있다.
- 이미 시작된 방에서는 다시 시작할 수 없다.
- 서버가 최종 참가자 목록을 고정한 뒤 게임을 생성한다.

### 3-3. 선 플레이어

1차 MVP에서는 서버가 참가자 중 한 명을 무작위로 선택한다.

```text
startingPlayerId = serverRandom(players)
```

추후 확장 시 타일 뽑기 방식의 선 결정 연출을 추가할 수 있다.

### 3-4. 초기 손패

각 플레이어에게 14개씩 무작위 분배한다.

```text
initialRackSize = 14
```

남은 타일은 공용 풀에 둔다.

### 3-5. 턴 방향

방 입장 순서를 기준으로 시계 방향 순환한다.

```text
2인: Player 1 → Player 2 → Player 1
3인: Player 1 → Player 2 → Player 3 → Player 1
4인: Player 1 → Player 2 → Player 3 → Player 4 → Player 1
```

## 4. 유효한 조합

테이블 위 모든 타일은 반드시 `RUN` 또는 `GROUP` 중 하나에 속해야 한다.

### 4-1. RUN

같은 색상의 연속된 숫자 3개 이상이다.

유효:

```text
RED 3, RED 4, RED 5
BLUE 9, BLUE 10, BLUE 11, BLUE 12
```

무효:

```text
RED 3, BLUE 4, RED 5
RED 3, RED 4
RED 3, RED 5, RED 6
RED 12, RED 13, RED 1
```

규칙:

- 최소 길이 3
- 색상이 모두 같아야 함
- 숫자가 1씩 증가해야 함
- 같은 숫자를 중복할 수 없음
- 1은 13 뒤에 이어질 수 없음
- 최대 길이는 13

### 4-2. GROUP

같은 숫자이면서 서로 다른 색상인 타일 3개 또는 4개다.

유효:

```text
RED 7, BLUE 7, BLACK 7
RED 10, BLUE 10, YELLOW 10, BLACK 10
```

무효:

```text
RED 7, BLUE 7
RED 7, RED 7, BLUE 7
RED 7, BLUE 8, BLACK 7
```

규칙:

- 길이 3 또는 4
- 숫자가 모두 같아야 함
- 색상은 중복될 수 없음
- 동일한 물리 타일은 한 번만 사용 가능

---

## 5. 첫 등록

### 5-1. 정의

아직 첫 등록을 완료하지 않은 플레이어가 자신의 손패만 사용해 테이블에 처음 타일을 내는 행동이다.

### 5-2. 점수 조건

이번 턴에 새로 내려놓은 하나 이상의 유효 조합 합계가 30점 이상이어야 한다.

```text
initialMeldScore >= 30
```

예:

```text
RED 10, RED 11, RED 12
점수 = 33
→ 성공
```

```text
RED 4, BLUE 4, BLACK 4
점수 = 12
→ 실패
```

여러 조합을 동시에 사용할 수 있다.

```text
RED 7, RED 8, RED 9 = 24
BLUE 2, YELLOW 2, BLACK 2 = 6
총점 = 30
→ 성공
```

### 5-3. 사용할 수 있는 타일

첫 등록에는 반드시 자신의 턴 시작 손패에 있던 타일만 사용한다.

금지:

- 기존 테이블 조합에 타일 추가
- 기존 테이블 타일 가져오기
- 기존 테이블 재조합
- 다른 플레이어 타일 사용

### 5-4. 첫 등록 턴의 추가 조작

첫 등록에 성공한 그 턴에는 첫 등록에 사용한 조합만 확정할 수 있다.

```text
첫 등록 성공
→ 같은 턴에 기존 테이블 조작 금지
→ 다음 자신의 턴부터 자유 조작 가능
```

구현과 판정의 명확성을 위해 이 규칙을 1차 MVP 기준으로 고정한다.

### 5-5. 조커 사용

첫 등록에 조커를 사용할 수 있다.

조커 점수는 조커가 대신하는 숫자다.

예:

```text
RED 10, JOKER(RED 11), RED 12
점수 = 33
```

조커의 대체 값은 서버가 전체 조합 문맥으로 확정한다.

### 5-6. 실패

30점 미만이거나 조합이 하나라도 무효라면 첫 등록 전체가 실패한다.

```text
부분 성공 없음
전체 원상복구
```

플레이어는 이후 타일을 1개 뽑고 턴을 종료하거나, 타일 배치를 취소하고 다른 배치를 시도할 수 있다.

---

## 6. 일반 턴

첫 등록을 완료한 플레이어는 다음 행동을 할 수 있다.

- 손패에서 새 조합 생성
- 기존 조합에 손패 타일 추가
- 기존 테이블 조합 분리
- 기존 테이블 조합 병합
- 기존 테이블 타일 이동
- 조커 교체 및 재사용
- 타일 뽑기
- 턴 확정
- 턴 취소

### 6-1. 최소 기여 조건

일반 턴을 확정하려면 자신의 손패에서 최소 1개 이상의 타일이 테이블로 이동해야 한다.

```text
rackToTableTileCount >= 1
```

기존 테이블 타일만 재정렬하고 손패를 줄이지 않는 행동은 확정할 수 없다.

### 6-2. 테이블 전체 유효성

턴 확정 시 특정 조합만 확인하지 않는다.

```text
테이블 위 모든 타일
→ 빠짐없이
→ 중복 없이
→ 각각 하나의 유효 RUN 또는 GROUP에 포함
```

조건을 만족해야 한다.

### 6-3. 수확한 타일

기존 테이블 조합에서 꺼낸 타일은 같은 턴 안에 다시 유효한 테이블 조합에 포함되어야 한다.

손패로 가져갈 수 없다.

---

## 7. 기존 테이블 재조합

### 7-1. 허용 예시

기존:

```text
BLUE 6, BLUE 7, BLUE 8, BLUE 9, BLUE 10
```

플레이어 손패:

```text
BLUE 8
```

결과:

```text
BLUE 6, BLUE 7, BLUE 8
BLUE 8, BLUE 9, BLUE 10
```

모든 조합이 유효하고 손패 타일을 최소 1개 사용했으므로 성공한다.

### 7-2. 네 번째 색상 추가 후 타일 분리

기존:

```text
RED 6, BLUE 6, YELLOW 6
```

손패:

```text
BLACK 6
```

결과:

```text
RED 6, BLUE 6, YELLOW 6, BLACK 6
```

그 후 하나의 타일을 꺼내 다른 유효 조합에 사용할 수 있다.

단, 턴 종료 시 모든 타일이 유효 조합에 속해야 한다.

### 7-3. 금지

- 테이블 타일을 손패에 저장
- 유효하지 않은 중간 상태를 확정
- 타일 유실
- 타일 복제
- 동일 tileId 중복 사용
- 첫 등록 전 재조합

---

## 8. 조커 규칙

조커는 유효 조합 안에서 필요한 하나의 일반 타일을 대신한다.

### 8-1. 조커 표현값

서버는 조커마다 현재 역할을 계산한다.

예:

```text
RED 5, JOKER, RED 7
→ JOKER = RED 6
```

```text
RED 9, BLUE 9, JOKER
→ JOKER = YELLOW 9 또는 BLACK 9 중 하나
```

GROUP의 조커가 둘 이상의 색상으로 해석될 수 있으면, 서버 내부에서는 조합을 유효하게 만드는 가능한 역할 중 하나를 결정한다. UI에는 숫자와 가능한 색상 문맥을 표시한다.

### 8-2. 첫 등록 전 회수 금지

플레이어가 자신의 첫 등록을 완료하지 않았다면 테이블의 조커를 회수하거나 이동할 수 없다.

### 8-3. 조커 교체

테이블의 조커가 대신하는 실제 타일로 교체할 수 있다.

교체 타일은 다음 중 하나에서 올 수 있다.

- 현재 플레이어 손패
- 기존 테이블의 다른 유효 조합

단, 최종 테이블 전체가 유효해야 한다.

### 8-4. GROUP의 조커 교체

3개짜리 GROUP에 조커가 있다면 빠진 색상 중 하나의 동일 숫자 타일로 교체할 수 있다.

예:

```text
RED 8, BLUE 8, JOKER
```

조커가 YELLOW 8 또는 BLACK 8 역할을 할 수 있으므로, 해당 가능한 색상의 8로 교체할 수 있다.

### 8-5. 회수한 조커 재사용

회수한 조커는 같은 턴에 새로운 유효 조합에 반드시 사용해야 한다.

금지:

```text
조커 회수
→ 손패에 저장
→ 턴 종료
```

### 8-6. 조커 포함 조합 재구성

조커가 포함된 조합도 타일 추가, 분리, 병합, 이동이 가능하다.

최종 상태가 유효하면 허용한다.

### 8-7. 승리와 조커

자신의 마지막 손패가 조커여도 유효한 조합에 내려놓아 손패가 0개가 되면 승리할 수 있다.

## 9. 타일 뽑기

### 9-1. 기본 행동

플레이어는 자신의 턴에 테이블 배치를 확정하지 않고 공용 풀에서 타일 1개를 뽑을 수 있다.

```text
drawCount = 1
```

### 9-2. 턴 종료

타일 뽑기는 즉시 턴을 종료한다.

뽑은 타일을 같은 턴에 사용할 수 없다.

### 9-3. 임시 조작 후 뽑기

플레이어가 테이블을 임시 조작한 뒤 타일 뽑기를 선택하면 먼저 턴 시작 상태로 원상복구한 후 1개를 뽑는다.

### 9-4. 공용 풀 고갈

공용 풀이 비어 있으면 타일을 뽑을 수 없다.

이때 플레이어가 유효한 행동도 할 수 없다면 `PASS` 처리할 수 있다.

1차 MVP에서는 자동 무승부 판정을 바로 하지 않고 연속 패스 횟수를 기록한다.

---

## 10. 턴 임시 상태

### 10-1. 턴 시작 스냅샷

각 턴 시작 시 서버는 다음 상태를 저장한다.

- 전체 공개 테이블
- 현재 플레이어 손패
- 타일별 소유 위치
- 첫 등록 완료 여부
- 공용 풀
- 게임 상태 버전
- 현재 플레이어

### 10-2. 임시 편집

드래그·배치 중 상태는 `TurnDraft`로 관리한다.

```text
확정된 GameState
+
현재 플레이어 전용 TurnDraft
```

다른 플레이어는 확정되지 않은 편집 결과를 볼 필요가 없다.

1차 MVP 기본 정책:

```text
다른 플레이어
→ 현재 플레이어가 "배치 중"임은 확인
→ 실제 테이블 변화는 턴 확정 후 수신
```

추후 실시간 구경 모드로 확장할 수 있다.

### 10-3. 턴 취소

턴 취소 시 `TurnDraft`를 폐기하고 턴 시작 스냅샷을 복원한다.

턴은 계속 유지되며 플레이어는 다시 행동하거나 타일을 뽑을 수 있다.

### 10-4. 확정 실패

서버 검증 실패 시:

- 확정 GameState는 변경하지 않음
- 실패 코드 반환
- TurnDraft는 유지하거나 사용자가 취소할 수 있음
- 다른 플레이어에게 상태 변경 이벤트를 보내지 않음

---

## 11. 턴 확정 검증 순서

서버는 다음 순서로 검증한다.

```text
1. 인증 사용자 확인
2. 현재 게임 참가자 확인
3. 현재 턴 확인
4. 요청 gameVersion 확인
5. tileId 존재 여부 확인
6. 타일 소유권 확인
7. 타일 중복·유실 확인
8. 첫 등록 규칙 확인
9. 최소 손패 기여 확인
10. 조커 규칙 확인
11. 전체 테이블 조합 유효성 확인
12. 손패 0개 여부 확인
13. 원자적으로 GameState 반영
14. gameVersion 증가
15. 플레이어별 상태 전송
16. 승리 또는 다음 턴 처리
```

중간 단계에서 실패하면 아무 상태도 확정하지 않는다.

---

## 12. 승리 판정

턴 확정 후 현재 플레이어 손패가 0개면 즉시 승리한다.

```text
rackSize == 0
→ GameStatus.FINISHED
→ winnerId = currentPlayerId
```

승리 후:

- 추가 게임 행동 차단
- 모든 참가자에게 결과 전송
- 게임 결과 DB 저장
- 방 상태를 `FINISHED`로 전환
- 재경기 또는 로비 이동 제공

---

## 13. 패스와 게임 종료 예외

### 13-1. PASS

공용 풀이 비어 있고 유효한 행동을 하지 못하는 경우 패스할 수 있다.

### 13-2. 연속 패스

모든 활성 플레이어가 한 바퀴 동안 연속 패스하면 교착 상태로 본다.

```text
consecutivePassCount >= activePlayerCount
```

1차 MVP 종료 기준:

- 각 플레이어 손패 점수 합계를 계산
- 가장 낮은 플레이어가 승리
- 동점이면 공동 승리 대신 `DRAW` 처리

이 규칙은 웹게임의 무한 반복을 막기 위한 프로젝트 운영 규칙이다.

### 13-3. 손패 점수

일반 타일:

```text
표시 숫자
```

조커:

```text
30점
```

## 14. 턴 제한시간

1차 MVP에는 서버 기반 턴 타이머 구조를 포함하되 기본값은 설정 가능하게 한다.

권장 기본값:

```text
turnTimeLimitSeconds = 120
```

시간 초과 시:

```text
턴 시작 스냅샷으로 전체 원상복구
→ 임시 배치와 재조합 결과 전부 폐기
→ 손패·테이블·조커 위치를 배치 전 상태로 복원
→ 공용 풀이 남아 있으면 1개 자동 뽑기
→ 풀이 비어 있으면 PASS
→ 다음 플레이어
```

중요:

- 유효한 배치를 일부 만들어 둔 상태라도 확정하지 않았다면 전부 취소한다.
- 잘못된 배치 때문에 시간이 초과된 경우에도 동일하게 턴 시작 상태로 돌아간다.
- 시간 초과 처리 중에는 다른 플레이어가 상태를 변경할 수 없다.
- 복원과 자동 뽑기 또는 PASS는 하나의 원자적 서버 처리로 수행한다.

개발 초기에는 타이머를 비활성화할 수 있다.

## 15. 접속 종료와 재접속

### 15-1. 일시 접속 종료

WebSocket 연결이 끊겨도 즉시 게임에서 제거하지 않는다.

```text
connectionStatus = DISCONNECTED
```

### 15-2. 재접속

동일 계정이 다시 접속하면:

- 기존 GamePlayer와 연결
- 자신의 손패 복원
- 공개 테이블 복원
- 현재 턴 복원
- 첫 등록 상태 복원
- 최신 gameVersion 제공

### 15-3. 현재 턴 플레이어 이탈

턴 타이머가 활성화되어 있으면 남은 시간을 계속 진행한다.

시간 초과 시 자동 뽑기 또는 PASS 처리한다.

### 15-4. 중복 접속

동일 계정의 게임 조작 연결은 하나만 활성화한다.

새 연결이 성공하면 이전 연결은 읽기 전용 또는 종료 처리한다.

- 3색 성공

- 숫자 불일치 실패
- 5개 실패

- 정확히 30 성공

- 여러 조합 합계 30 성공

- 조커 포함 성공

- 성공 턴에 기존 테이블 조작 실패

- 긴 RUN 분리 성공

- 테이블 전체 중 하나라도 무효면 실패
- 기존 타일 손패 이동 실패
- 손패 타일 미사용 실패

### 조커

- RUN 대체 성공
- GROUP 대체 성공
- 올바른 타일로 교체 성공
- 잘못된 값 교체 실패
- 회수 후 같은 턴 재사용 성공
- 회수 후 미사용 실패
- 첫 등록 전 회수 실패

### 턴과 원자성

- 현재 턴 외 요청 실패
- 중복 확정 한 번만 성공
- 오래된 gameVersion 실패
- 유효하지 않은 확정 후 상태 불변
- 취소 후 스냅샷 동일
- 시간 초과 후 턴 시작 스냅샷 복원

- 드로우한 타일 같은 턴 사용 불가

### 보안

- 상대 손패 조회 불가

- 풀 고갈 및 전체 패스 종료

## 18. 구현 시 핵심 불변조건

어떤 요청 전후에도 다음 조건이 유지되어야 한다.

```text
모든 106개 tileId는 정확히 한 위치에 존재한다.
```

가능한 위치:

- 공용 풀
- 특정 플레이어 손패
- 공개 테이블

추가 조건:

```text
한 tileId가 두 위치에 동시에 존재하지 않는다.
tileId가 사라지지 않는다.
테이블의 확정 타일은 모두 유효 조합에 포함된다.
현재 턴 플레이어만 GameState를 변경한다.
실패한 요청은 확정 상태를 변경하지 않는다.
상대 손패는 상대에게 전송하지 않는다.
```

## 21. 게임 모드별 규칙

### 21-1. CLASSIC

```text
첫 등록: 30점 이상
턴 제한: 기본 120초
승리 조건: 손패 0개
교착 종료: 최저 손패 점수 승리
랭킹 반영: 예
```

교착 종료 시 최저 손패 점수가 같은 플레이어가 둘 이상이면 `DRAW` 처리한다.

### 21-2. SPEED

```text
전체 게임 제한: 5분
턴 제한: 기본 20초
첫 등록: 제한 없음
승리 조건: 5분 종료 후 최종 점수
랭킹 반영: 아니오
```

스피드 점수:

```text
자신의 손패에서 테이블에 확정한 타일 점수
- 종료 시 남은 손패 점수
```

- 일반 타일은 표시 숫자만큼 계산한다.
- 테이블에 확정한 조커는 대체 숫자만큼 계산한다.
- 손패에 남은 조커는 30점 감점한다.
- 기존 테이블 타일을 이동하는 행위는 추가 점수를 만들지 않는다.
- 각 타일은 최초로 손패에서 테이블에 확정한 플레이어의 기여 점수로 기록한다.
- 잘못된 배치, 턴 취소, 시간 초과 롤백 시 해당 점수 변화도 함께 롤백한다.

전체 5분 종료 시:

```text
현재 미확정 배치 롤백
→ 추가 드로우 없음
→ 즉시 게임 종료
→ 최종 점수 계산
```

최고 점수가 같은 플레이어가 둘 이상이면 `DRAW` 처리한다.

## 22. 랭킹 점수 규칙

### 22-1. 기본 정책

- CLASSIC 결과만 반영한다.
- SPEED 결과는 반영하지 않는다.
- 초기 랭킹 점수는 1000점으로 시작한다.
- 한 경기의 상대 기준 점수는 자신을 제외한 참가자들의 경기 시작 전 평균 점수다.
- 점수 계산은 경기 종료 시 서버에서 한 번만 수행한다.

### 22-2. 기대 승률

권장 계산식:

```text
expectedScore = 1 / (1 + 10 ^ ((opponentAverageRating - playerRating) / 400))
```

실제 결과값:

```text
승리 = 1.0
DRAW = 0.5
패배 = 0.0
```

변동값:

```text
ratingDelta = round(K × (actualScore - expectedScore))
```

기본 K 값:

```text
K = 32
```

예시:

```text
내 점수 1000
상대 평균 800
승리
→ 기대 승률이 높으므로 적은 점수 획득

내 점수 1000
상대 평균 1000
승리
→ 보통 수준 점수 획득

내 점수 1000
상대 평균 1200
승리
→ 기대 승률이 낮으므로 많은 점수 획득
```

### 22-3. 다인전 처리

2~4인 CLASSIC 경기에서도 각 플레이어는 자신을 제외한 참가자 평균 점수를 상대 기준으로 사용한다.

```text
opponentAverageRating
= 다른 참가자들의 시작 랭킹 점수 합계 / 다른 참가자 수
```

결과값:

```text
단독 승리자 = 1.0
DRAW 참가자 = 0.5
패배자 = 0.0
```

교착 종료에서 한 명이 최저 점수로 승리하면 그 사용자만 승리로 처리한다.

### 22-4. 최소 점수

랭킹 점수는 0점 아래로 내려가지 않는다.
